# /stock-trend 报告可执行性优化设计

**日期**: 2026-05-29  
**范围**: `.claude/skills/stock-trend` 的 `/stock-trend` 单标的报告（Markdown + HTML）  
**目标**: 在不推翻现有报告骨架的前提下，显著提升“现在能不能买、买在哪、什么时候减仓/止盈”的可执行性。

## 1. 背景

当前 `/stock-trend` 报告已经具备完整分析信息：结论、关键信号、风险提示、操作建议区间、监控信号、K 线形态、综合研判。

问题不在于信息不够，而在于**执行信息分散**：

1. 用户需要先读完整个报告，才能拼出“今天该不该动”的结论。
2. “操作建议区间”已有支撑/压力/止损/目标，但缺少明确的双场景执行语义。
3. 减仓、止盈、观望这些动作没有被标准化表达，阅读成本偏高。

本次优化聚焦在**基于现有格式增强**，不做整份报告的大改版。

## 2. 设计目标

本次设计只解决以下问题：

1. 报告开头能更快回答“今天该不该动”。
2. 操作建议区能同时覆盖“继续上冲”和“回调到位”两种场景。
3. 减仓、止盈、失效条件写成可执行动作，而非仅提供静态价位。
4. Markdown 与 HTML 输出口径保持一致。

## 3. 非目标

以下内容不在本次设计范围内：

1. 不重做整份报告的信息架构。
2. 不新增全新的量化评分体系。
3. 不改变 `/etf-scan`、`/longtou` 等其他报告的输出结构。
4. 不引入盘中盯盘型、分钟级交易建议。

## 4. 方案对比

讨论阶段比较过 3 条路径：

1. **保守增强（最终选择）**：保留现有章节顺序，只增强“结论”和“操作建议区间”。
2. 轻量前置摘要：在现有结构前增加操作摘要卡。
3. 场景化重写操作区：把操作区改成三栏作战面板。

最终选择方案 1，原因：

1. 最符合“基于现有报告格式优化”的约束。
2. 用户最关心的是执行性，而不是视觉重构。
3. 对现有模板、黄金快照和阅读习惯的冲击最小。

## 5. 最终设计

### 5.1 总体结构

报告主骨架保持不变：

1. 标题
2. 结论
3. 关键信号
4. 风险提示
5. 操作建议区间
6. 监控信号
7. K 线形态
8. 综合研判

仅做两处增强：

1. 在结论下方新增一行 **今日动作**。
2. 将 **三、操作建议区间** 升级为更明确的执行区，但不改变它在整份报告中的位置。

### 5.2 今日动作

在现有结论区域中增加标准化动作语句，用一句话回答“今天该不该动”。

建议动作标签限定为以下集合：

1. `可低吸`
2. `等回踩`
3. `只观察`
4. `分批止盈`
5. `等待突破确认`
6. `已有仓位继续持有`

设计原则：

1. 文案要明确，但不写成绝对命令。
2. 动作是执行提示，不替代下方详细价位。
3. 当信号不足时，优先输出保守动作，而不是勉强给买点。
4. `不追高` 作为解释性短语附着在 `等回踩` 或 `只观察` 后，不单独作为一级动作标签。

示例：

> 今日动作：等回踩，不建议在当前位置追高；若回踩至 24.8-25.2 可分两笔试仓。

### 5.3 操作建议区升级

保留现有支撑位、压力位、当前价、止损位、目标价位等字段，但新增 4 个固定模块。

#### A. 今日动作判定

用一句话概括当前最优动作，作为操作区的总标题或首行摘要。

#### B. 双场景执行

固定输出两个场景：

1. **场景 A：继续上冲**
   - 是否允许追高
   - 若允许，需要满足什么条件（如放量突破、站稳关键位）
   - 若不允许，直接写“只观察，不追”

2. **场景 B：回调到位**
   - 给出入场区间
   - 指出是否建议分两笔建仓
   - 说明对应止损位和最大容错

这样可以覆盖用户最关心的两类真实决策：现在追不追、回调买不买。

#### C. 退出与减仓规则

将现有目标价升级成**价位 + 动作**的组合表达：

1. 到达 TP1 → 减仓 1/3
2. 到达 TP2 → 再减仓 1/3
3. 剩余仓位 → 视趋势强弱继续持有或保护止盈
4. 跌破止损位 → 视为计划失效

关键点：目标位不再只是静态展示，而是直接绑定动作。

#### D. 执行时间窗

新增一个简短的有效期说明，避免报告变成无限期建议。

示例：

1. 未来 5-10 个交易日有效
2. 若财报披露前未触发买点，则本计划失效
3. 若重要会议或事件临近，等待事件落地后重评

### 5.4 判定逻辑

不新增独立评分系统，直接复用现有报告已生成的数据字段：

1. 综合方向
2. 综合评分
3. 置信度
4. 当前价 / 支撑位 / 压力位 / 止损位
5. TP1 / TP2 / 风险收益比
6. 关键事件时间窗（如已有）

在此基础上增加 4 条动作护栏：

1. **低置信度或风险收益比不足**  
   默认偏向 `只观察` 或 `等回踩`。

2. **当前位置接近支撑、止损空间合理**  
   才允许给出 `可低吸` 或 `回踩试仓`。

3. **当前位置接近压力或 TP1/TP2**  
   优先给出 `等回踩（不追高）`、`分批止盈` 或 `已有仓位继续持有`。

4. **临近关键事件**  
   自动收敛动作表达，强调等待确认，避免给出过激进的执行建议。

### 5.5 Markdown 与 HTML 的一致性

两种输出都要包含相同的信息槽位：

1. 今日动作
2. 双场景执行
3. 退出与减仓规则
4. 执行时间窗

差异只允许体现在呈现层：

1. HTML 可以用卡片、强调色、分区来提升扫读效率。
2. Markdown 仍以表格和短段落为主，保证终端可读性。
3. 不允许 HTML 有额外决策信息而 Markdown 缺失。

### 5.6 降级与异常处理

当关键数据不足时，报告必须**显式降级**，而不是继续伪装成高确定性建议。

规则：

1. 缺少 K 线或关键技术位时，不输出过细的入场动作。
2. 缺少风险收益比或目标位时，不输出完整的追高/回调执行计划。
3. 缺少关键事件信息时，可省略时间窗，但不能编造事件。
4. 数据不足时，统一偏向保守结论，如 `只观察`、`等待更多确认`。

## 6. 影响范围

核心影响面预计包括：

1. `assets/report-template.md`
2. `assets/report-template.html`
3. `scripts/generate_report.py`
4. 与 `/stock-trend` 报告相关的测试与 golden snapshots

本设计不要求调整 ETF 扫描或龙头报告模板。

## 7. 验证要求

至少覆盖以下 5 类典型场景：

1. 多头但已偏离理想买点 → `等回踩（附注：不追高）`
2. 多头且接近支撑 → `可低吸 / 回踩试仓`
3. 价格接近 TP1 或 TP2 → `分批止盈`
4. 低置信度或关键事件临近 → `只观察`
5. 数据缺失 → 保守降级
验证目标不是只看模板能否渲染，而是确认生成的动作文案与已有评分、价位逻辑一致。

## 8. 成功标准

如果本次优化成功，用户应能在 30 秒内从报告中直接读出：

1. 今天该不该动
2. 若动，应该等上冲确认还是等回调到位
3. 何处止损
4. 何处分批减仓或止盈
5. 该建议有效到什么时候

---

本设计遵循“**不改骨架，只增强执行性**”的原则，优先服务中线波段用户的真实决策路径，而非增加更多分析噪音。
